# 🧠 SQL for ML — Feature Engineering (SQL) — 01_feature_engineering

This module covers **feature engineering patterns in SQL**
for machine learning pipelines.

Before modeling begins, raw transactional or profile data must be transformed into
**model-ready features**.

SQL is often the best place to do this because it provides:

- reproducibility
- warehouse-scale transformation
- close access to source data
- stable feature generation logic

본 디렉토리는 머신러닝 파이프라인을 위한  
**SQL 기반 feature engineering 패턴**을 다룹니다.

모델링 이전에는 원시 데이터(raw data)를 그대로 사용하는 것이 아니라,  
의미 있는 **파생 변수(feature)** 로 변환해야 합니다.

SQL은 특히 다음 장점 때문에 feature engineering에 강력합니다:

- 재현 가능한 변환 로직
- 대용량 데이터 처리
- 원천 데이터와 가까운 위치에서 처리 가능
- feature table / feature mart 구축에 적합

## ⚙️ Core SQL for ML Principles

### 1️⃣ Features should be reproducible

A feature must be generated by a deterministic SQL rule.

feature는 사람이 임의로 만드는 값이 아니라,
항상 같은 입력에 대해 같은 결과를 만들어야 합니다.

---

### 2️⃣ Use the warehouse as the source of truth

When possible, feature logic should live close to source data.

가능하면 SQL 기반 feature logic은
원천 데이터와 가까운 데이터웨어하우스 계층에서 관리하는 것이 좋습니다.

---

### 3️⃣ Aggregation is one of the strongest baselines

Simple grouped features often provide strong modeling signals.

복잡한 모델보다 먼저,
좋은 aggregation feature가 더 강한 baseline이 되는 경우가 많습니다.

---

### 4️⃣ Encoding choice depends on the model and data shape

There is no single best categorical encoding.
	•	low-cardinality → one-hot often works well
	•	high-cardinality → frequency / grouped categories may be safer
	•	tree models vs linear models → encoding sensitivity differs

---

### 5️⃣ Feature tables should be model-ready

The end goal is not a clever query.
The goal is a clean, reusable feature dataset.

최종 목적은 복잡한 SQL 자체가 아니라,
모델이 바로 사용할 수 있는 재사용 가능한 feature table입니다.
